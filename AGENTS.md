# AGENTS.md

本文档用于约束在 `web-presentation` 仓库内工作的开发者与智能代理行为，目标是保证平台规划、工程实现与 `runtime` 子项目协作保持一致。

## 1. 基础协作规则

- 使用中文沟通、提交说明与文档描述。
- 不要把单个文件写得过大；超过 1000 行时优先考虑拆分。
- 每个新增源码文件的开头都要有“文件功能描述”注释。
- 为函数补充中文注释，重点说明用途、参数、返回值与关键副作用。
- 前端依赖统一使用 `pnpm`。
- Python 依赖统一使用 `uv`，虚拟环境统一使用 `venv`。

## 2. 仓库定位

这是一个平台集成仓库，不只是单纯的前端项目。

仓库目标包括：

- 维护平台级架构设计与模块边界
- 管理 `backend`、`editor` 与 `runtime` 的协作方式
- 沉淀控制面与数据面的接口契约
- 承载 Docker 编排、环境变量规范与部署集成说明

当前需要特别注意的是：`runtime/` 是独立项目 `PPT-Engineering` 的子模块接入目录，而不是本仓库内部普通子目录；它需要同时兼顾独立项目形态和平台运行时形态。

## 3. 目录职责

### 顶层目录

- `README.md`：面向项目成员，说明平台目标、架构、流程、模块边界与协作方式。
- `AGENTS.md`：面向开发者与智能代理，说明本仓库的修改约束与工作规则。

### backend/

用于承载 Backend 控制面服务。未来实现应聚焦：

- 文件与配置持久化
- 历史版本与快照管理
- Runtime 调度
- 构建任务状态管理
- 产物接收与托管

### editor/

用于承载 Editor 工作台。未来实现应聚焦：

- 代码编辑
- 文件树与组件选件
- Iframe 预览
- 构建操作入口
- Runtime 重启态感知

### runtime/

这是独立项目 `PPT-Engineering` 在当前仓库中的接入目录，建议以双模式方式维护：

### standalone 模式

负责：

- 运行时预览
- 本地文件管理
- 可视化编辑
- 基于 Vite 的本地 HMR 与构建

### platform-runtime 模式

负责：

- HMR 驱动
- Backend 同步后的受控写盘
- 动态构建
- 构建产物压缩与回传
- 内部 API 与容器自愈链路

修改该目录前，必须先理解它的独立性与同步方式。

### 命名约定

- `PPT-Engineering`：指独立项目名称。
- `runtime/`：指 `PPT-Engineering` 在当前仓库中的目录路径。
- `Runtime`：指平台架构中的运行时角色。
- `platform-runtime`：指 `PPT-Engineering` 面向平台接入时的运行模式。

## 4. Runtime 子模块修改规则

### 原则

- 将 `runtime/` 视为独立项目 `PPT-Engineering` 的接入目录，不是任意重构的普通目录。
- 平台仓库只负责集成、引用和协议协同，不应在不考虑上游的情况下随意改坏 Runtime 的独立可用性。
- 如果某项能力本质属于 Runtime 底座，应优先在 Runtime 自身语境下设计与实现。
- 独立项目能力默认保留，平台能力应通过模式切换接入，而不是直接替换现有使用方式。

### 推荐做法

1. 涉及 `PPT-Engineering` 内部实现时，优先查看 `runtime/README.md` 和 `runtime/AGENTS.MD`。
2. 修改前，先判断当前需求属于 `standalone` 还是 `platform-runtime`。
3. 如果修改仅影响 `PPT-Engineering` 内部页面、组件、工具或构建逻辑，应按子项目规范进行。
4. 如果修改影响 Backend / Editor 与 Runtime 的交互协议，必须同步更新顶层文档。
5. 如果 `PPT-Engineering` 上游已有独立仓库演进节奏，应优先在上游完成变更，再同步本仓库子模块提交。
6. 新增平台能力时，优先增加模式开关、适配层和能力抽象层，不要把平台逻辑直接散落进现有编辑器或设置面板。

### 禁止事项

- 不要把平台特有的临时说明直接塞进 Runtime 内部文档，除非它确实属于 Runtime 的长期能力。
- 不要假设 Runtime 可以长期承载 Backend 的控制逻辑。
- 不要在顶层文档里复制大段 Runtime 内部实现细节，避免双份文档失真。
- 不要为了平台接入直接删除 `PPT-Engineering` 现有的文件管理、编辑器、资源管理等独立项目能力。
- 不要继续把本应属于 `editor/` 的平台工作台能力无边界堆进 Runtime。

## 5. 文档维护规则

### 什么时候更新顶层 README

出现以下情况时，应同步更新顶层 `README.md`：

- 平台愿景、模块职责或架构关系发生变化
- Backend、Editor、Runtime 的边界重新划分
- 内部关键流程发生变更
- 子模块协作方式、部署方式或开发入口改变

### 什么时候更新顶层 AGENTS

出现以下情况时，应同步更新顶层 `AGENTS.md`：

- 团队协作规范发生变化
- 新增统一编码约束或文档约束
- 子模块修改策略变化
- Monorepo 结构新增或调整

### 什么时候更新 Runtime 文档

出现以下情况时，应优先更新 `PPT-Engineering` 自身文档：

- `PPT-Engineering` 新增开发命令、目录结构或基础能力
- `PPT-Engineering` 新增或调整模式切换策略
- `PPT-Engineering` 插件系统、构建机制、页面体系发生变化
- `PPT-Engineering` 内部编码规范、页面开发方式发生变化

## 6. 实现建议

在本仓库推进工程化时，建议遵循以下顺序：

1. 先定义 Backend 与 Runtime 的内部接口契约。
2. 再为 `PPT-Engineering` 增加 `runtimeMode` 与能力开关层。
3. 然后补齐 Docker Compose 与环境变量设计。
4. 再初始化 Backend 骨架与数据库模型。
5. 再初始化 Editor 骨架并打通预览链路。
6. 最后再补充构建导出、产物托管与故障恢复闭环。

## 7. 编码注意事项

- 新增 Python 代码时，优先采用清晰的模块拆分，不要把路由、模型、服务全部写入单文件。
- 新增前端代码时，优先拆分视图、组件、状态逻辑和 API 请求层。
- 如果接口契约尚未稳定，先在文档中声明请求路径、入参、出参和错误语义，再开始跨模块联调。
- 如果需要为平台新增共享约束，优先写入顶层文档，而不是只留在会话记录里。
- 涉及 Runtime 平台化改造时，优先新增模式判断、能力矩阵、适配层目录，不要把 `standalone` 与 `platform-runtime` 的行为直接硬编码在同一套页面逻辑里。

## 8. 交付期望

在这个仓库中完成任务时，默认交付应尽量满足以下要求：

- 改动与平台规划一致
- 不破坏 `PPT-Engineering` 子模块的独立性
- 不破坏 `PPT-Engineering` 的 `standalone` 可用性
- 文档与代码同步更新
- 能明确区分“平台职责”和“Runtime 职责”
- 能明确区分 `PPT-Engineering` 的独立项目能力和平台模式能力
- 后续接手人只看顶层文档也能理解仓库边界
