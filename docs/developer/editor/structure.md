# 前端结构约定

Editor 代码应按视图、组件、组合式逻辑、状态和 API 请求层拆分，避免单个文件同时承担页面布局、请求、状态转换和业务规则。

## 视图

视图负责路由级页面结构和主要布局。复杂视图应把区域拆成子组件，把数据请求、状态派生和表单逻辑移到组合式逻辑或 store。

## 组件

通用 UI 优先复用 `components/ui`，业务组件优先放在已有领域目录，例如 `components/project`、`components/agent`。新增组件应保持输入输出清晰，不直接依赖无关页面状态。

## 组合式逻辑

组合式逻辑适合封装表单状态、API 调用、轮询、确认流程和局部业务状态。跨页面共享逻辑应保持命名稳定，并配套测试。

## API 请求层

请求函数应集中维护，返回结构与 Backend schema 对齐。Backend 契约变化时，同步更新请求层、类型和相关测试。

## 状态管理

状态应按业务域组织。避免把临时 UI 状态、服务端事实源和派生状态混在一个对象中，减少刷新和会话恢复时的错位。
